Core Data Model & Concepts

A deep dive into the AWA ontology. Understand every entity, its purpose, and how to model high-performance agentic workflows.

1. The Workflow

The Workflow is the root entity. It defines the boundaries of a specific business process, its versioning, and its high-level objectives.

Best Usage:

Keep workflows focused on a single outcome (e.g., "Claims Settlement" rather than "Insurance Operations"). Use semantic versioning (e.g., 1.2.0) to manage evolution.

2. Activities & Units of Work

An Activity represents a single logical step in the process. Unlike traditional BPM, AWA activities are strictly typed by their intended performer.

Concept Description Best Practice
Activity A task performed by an actor. Atomic, idempotent where possible, with clear procedures.
Decision A gateway for routing based on logic. Use for branching. Map logic to specific queryable context fields.
Event Tiggers or milestones. Use evt-start and evt-end to demarcate boundaries.

3. Actors & Roles

The performant heart of AWA. We distinguish between Who (Role) and What (Actor Type).

ai_agent

Autonomous LLM-driven entities. Best for cognition, triage, and extraction.

human

Human collaborators. Best for oversight, exception handling, and high-value authorization.

application

Traditional software systems. Best for integrations, batch jobs, and deterministic tasks.

4. Contextual State

The Context is how data flows. Instead of complex message payloads, actors interact with shared context variables.

Sync Pattern: Shared State is the most common. It allows multiple agents to contribute to a single "source of truth" (like a claim record) as the workflow progresses.

Usage Best Practice:

  • Minimalism: Don't put everything in one context. Group by functional area (e.g., customer_data, claim_data).
  • Access Control: Always define AccessRights (read/write) per activity to prevent context corruption.

5. Edges & Transfer Types

Edges connect activities. In AWA, edges also describe the Nature of Handoff via transfer_type.

  • integration_flow: System-to-system or AI-to-System. Automated and instantaneous.
  • manual_work: Human-to-Human or Human-to-AI. Higher latency, requires monitoring.
  • notification: Asynchronous inform-only handoffs.

6. Execution Metrics (Performance)

Data concepts for time and efficiency.

  • duration_ms: The "Design-Time" expected duration.
  • bottleneck_threshold_ms: The "Tolerance" before a waste category is triggered.

Tip: Always set thresholds. This fuels the 3D visualization's glowing bottleneck indicators.

7. Auxiliary Concepts

Technology

Describes the underlying stack (e.g., database, rest_api) used by actors.

Collection

A group of related workflows, providing a catalog-level view of processes.

SLA

Service Level Agreements that define the bound for acceptable performance.
